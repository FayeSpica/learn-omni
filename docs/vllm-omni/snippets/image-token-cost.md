# 图片 encode 占多少上下文：以 Qwen3-Omni 为例，对比文本

> **一句话结论**：图片占上下文 = 占 token，且是**确定性可算**的。Qwen3-Omni 视觉分支沿用 Qwen2.5-VL/Qwen3-VL 的 patch(14)+merge(2×2)，**1 个视觉 token = 原图 28×28 像素**。直觉换算：**看一张普通照片 ≈ 读一页文档**；长边翻倍 → token 变 4 倍（二次增长）。

## 核心公式

视觉编码器关键参数：

- `patch_size = 14`：每个 patch 覆盖 14×14 像素
- `spatial_merge_size = 2`：相邻 2×2 个 patch 合并成 1 个 LLM token

所以 **1 个视觉 token 对应原图 28×28（=14×2）像素的区域**。

图片先经 `smart_resize` 把宽高对齐到 28 的倍数、并夹到 `[min_pixels, max_pixels]` 区间，然后：

```
tokens ≈ ⌈H / 28⌉ × ⌈W / 28⌉
```

外面再包一层 `<|vision_start|> … <|vision_end|>`，**+2 个 token** 的固定开销。

默认区间：

- `min_pixels = 4 × 28 × 28 = 3136` → 单图最少 **4** token
- `max_pixels = 16384 × 28 × 28 ≈ 12.8M` → 单图最多 **16384** token

即单图 token 被夹在 **[4, 16384]**。服务端通常把 `max_pixels` 调更小来省显存/上下文，等价于强制降采样。

## 常见分辨率的 token 数

| 图片 | 对齐后网格 | 视觉 token | 相当于文本 |
|---|---|---|---|
| 224×224（缩略图）| 8×8 | **64** | ~一句话 |
| 448×448 | 16×16 | **256** | ~一段（180 英文词 / 350 汉字）|
| 1280×720（截图/照片）| 46×26 | **≈1196** | ~一页 A4 |
| 1024×1024 | 37×37 | **≈1369** | ~一页 |
| 1920×1080（1080p）| 69×39 | **≈2691** | ~两页 |
| 4032×3024（手机原图）| 被 `max_pixels` 夹住 | **≤16384** | ~一篇长文 |

## 和文本对比

- **文本**：英文 1 token ≈ 0.75 词；中文 1 token ≈ 1~1.5 字。一段普通中文（约 300 字）≈ 200~300 token。
- **图片**：一张 448×448 的小图 ≈ **256** token，就已吃掉一整段文字的预算；一张 1080p 图 ≈ **2700** token，约等于两页纯文字。

关键直觉：**图片 token 对分辨率是二次增长**——H、W 各自线性，长边翻倍 token 就变 4 倍。这也是 VLM 里图片经常是上下文大头的原因。

## Omni 场景的两点提醒

1. **视频更凶**：Qwen3-Omni 视频按帧抽样，每帧都是一张图，再叠 `temporal` 维度。例如 2fps 抽 30s = 60 帧，每帧哪怕压到 256 token，也是 **~1.5 万 token**——视频 token 预算基本由 `帧数 × 每帧分辨率` 决定，很容易顶爆上下文。控制手段：压 `max_pixels`（降每帧分辨率）+ 降抽帧率。相关时间对齐见 [P2 图/视频→ViT→TMRoPE](../vision-tmrope-path.md)。
2. **省 token 的旋钮**：服务端/processor 调小 `max_pixels`（或设 `min_pixels`）是最直接的，等价于强制降采样；对纯识别类任务通常掉点很小，但 token 能砍一半以上。

## 一分钟自测

```
tokens ≈ (对齐到28倍数的高 / 28) × (对齐到28倍数的宽 / 28) + 2
```

拿到实际请求的 `max_pixels` 配置后，把长宽先夹进区间再套公式，就能估出单张图在这次请求里占多少上下文、占比多大。
